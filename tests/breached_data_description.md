# Пояснение к синтетической базе утечек 
## 1. Источник данных
База сформирована на основе **популярных паролей из открытых источников**
## 2. Какие пароли включены
В синтетическую базу добавлены пароли, которые часто используются пользователями и считаются небезопасными.  
Примеры паролей из базы:
- `qwerty123`
- `admin123`
- `qwertyuiop`
- `123456789`
- `abc123`
- `password1`
- `654321`
## 3. Как пароль превращается в хэш
Пароли **не хранятся в открытом виде**. В файле базы данных `breached_hashes.db` хранятся только их хэши. В проекте используется алгоритм SHA-256.
Функция `hash_password` (файл `backend/utils/hash.py`) работает так:
1. Пароль преобразуется в байты (UTF-8).
2. Вычисляется SHA-256 хэш.
3. Хэш возвращается в виде строки из 64 шестнадцатеричных символов.
## 4. Как проверяется пароль на утечку
1. Пользователь вводит пароль в форме.
2. Бэкенд вычисляет SHA-256 хэш этого пароля (функция `hash_password`).
3. Вычисленный хэш ищется в таблице `breached_hashes`.
4. Если хэш найден — сервер возвращает ответ:
   ```json
   {
     "strength": "weak",
     "reasons": ["Обнаружен в базе утечек"]
   }
   ```
## 5. Где находится база в проекте
- Файл базы данных: `backend/data/breached_hashes.db`
- Модель таблицы: `backend/models/passwords.py` (класс `BreachedHash`)
- Логика проверки утечек: `backend/utils/check_breached_password.py`
- Функция хэширования: `backend/utils/hash.py`
## 6. Как повторить генерацию (инструкция)
Если потребуется обновить или пересоздать базу утечек:
1. Взять список популярных паролей (например, из `rockyou.txt` или собранный вручную).
2. Для каждого пароля вычислить SHA-256 хэш.
3. Загрузить хэши в таблицу `breached_hashes`.
## 7. Связь с тестированием
В автотестах (`автотесты.py`) есть проверки:
- Пароли из базы → ожидается `"Обнаружен в базе утечек"` 
- Пароли не из базы → утечек нет 
### Примеры из автотестов
```python
("qwerty123", "weak", True),        # из базы, утечка есть
("StrongP@ss2025", "strong", False) # не из базы, утечки нет
```
## 8. Безопасность

- Исходные пароли **не хранятся**, только хэши.
- Хэш невозможно обратить в пароль (SHA-256 — односторонняя функция).
- База не содержит персональных данных пользователей.

## 9. Как оценивается сложность пароля (алгоритм)

Логика проверки находится в файле `backend/utils/check_strength.py`:

1. **Пустой пароль** → `weak` + сообщение «Введите пароль»
2. **Обнаружена утечка** → `weak` + «Обнаружен в базе утечек» (дальше не проверяется)
3. **Длина** (<8 → сразу `weak`; 8–11 → +1 балл; 12+ → +2 балла)
4. **Состав** (+1 за цифры, заглавные, спецсимволы; –1 за только цифры или только буквы)
5. **Запрещённые последовательности** (–1 за `123`, `qwerty`, `password`, повторы, годы)
6. **Итоговая категория** (0–1 → `weak`; 2–3 → `medium`; 4+ → `strong`)

## 10. Примечание
База сформирована на основе **реальных популярных паролей**. Пароли не выдуманы искусственно, а взяты из реальных списков часто используемых паролей.  

Синтетическая база утечек создана **для учебного проекта**, не содержит персональных данных пользователей и используется только для демонстрации работы функционала проверки утечек.
