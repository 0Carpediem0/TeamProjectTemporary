# Отчёт по 2 итерации проекта

## Общая информация
Во второй итерации команда продолжила разработку сервиса проверки стойкости паролей и их присутствия в базах утечек. Основной целью этапа было собрать рабочее MVP, синхронизировать backend, frontend и тестирование, а также оформить результаты работы в репозитории и сопроводительной документации.

По итогам итерации проект был доведён до состояния рабочего прототипа: реализована серверная логика проверки пароля, подготовлен пользовательский интерфейс, настроено взаимодействие между фронтендом и бэкендом, проведено тестирование и собраны артефакты.

## Выполненные работы

### Backend
Во второй итерации backend-часть была существенно развита. Были выполнены следующие задачи:

- составлена схема хранения хэшей паролей;
- реализован алгоритм оценки стойкости пароля;
- написан POST-эндпоинт `/api/check`;
- подготовлен скрипт для генерации синтетического набора хэшей для тестирования;
- обеспечена связка backend-логики с фронтендом.

В результате backend стал основой MVP и обеспечил проверку пароля по длине, составу, запрещённым последовательностям и наличию в базе утечек.

### Frontend
Во второй итерации frontend был доведён до рабочего пользовательского интерфейса. Были выполнены следующие задачи:

- переработан дизайн интерфейса;
- обновлена цветовая схема;
- добавлен pop-up для отображения результата проверки;
- реализованы светлая и тёмная темы;
- настроен `fetch` для получения данных от backend;
- нормализована работа API на клиентской стороне.

Фронтенд стал полноценной точкой взаимодействия пользователя с сервисом и обеспечил получение и отображение результата проверки в удобной форме.

### Тестирование
Во второй итерации была проведена системная проверка работоспособности проекта. Тестировщик выполнил:

- ручное тестирование API через Postman;
- проверку backend по чек-листу безопасности;
- проверку frontend;
- оформление отчёта по тестированию;
- фиксацию найденных багов;
- сбор артефактов тестирования.

Отдельно были подготовлены тест-кейсы и чек-лист безопасности, что позволило организовать тестирование более последовательно и зафиксировать результаты в воспроизводимом виде.

## Результаты тестирования
В качестве основы для итоговой оценки использовался отчёт по тестированию за второй спринт. В ходе проверки тестировались:

- backend API `/api/check`;
- безопасность сервиса, включая логи, базу данных и работу методов POST/GET;
- frontend, включая поле ввода, консоль браузера и `localStorage`.

Тестирование проводилось вручную через Postman, через просмотр серверных логов и через инструменты разработчика в браузере.

По итогам тестирования были подтверждены следующие результаты:

- корректно определяется слабый пароль `1234567`;
- корректно определяется средний пароль `Pass1234!`;
- корректно определяется сильный пароль `StrongP@ss2025`;
- GET-запрос к API запрещён и возвращает `405 Method Not Allowed`;
- проверка утечек работает корректно;
- поле ввода пароля скрывает вводимые символы;
- пароль не отображается в консоли браузера;
- пароль не сохраняется в `localStorage`;
- пароль не выводится в серверных логах;
- в базе данных хранятся только хэши.

По результатам тестирования был обнаружен один значимый баг: при отправке пустого пароля сервер возвращает `weak`, хотя корректным поведением должна быть ошибка валидации `400 Bad Request`.

## Основные итоги итерации
По совокупности выполненных задач можно сделать вывод, что во второй итерации команда успешно собрала рабочее MVP сервиса. Были реализованы ключевые компоненты проекта:

- backend-логика оценки стойкости пароля;
- механизм проверки по базе хэшей;
- пользовательский интерфейс;
- интеграция frontend и backend;
- ручное тестирование и оформление результатов.

Также была организована рабочая структура репозитория и собрана сопроводительная документация по итогам итерации.

## Трудности и слабые места
Рефлексии участников показывают, что в ходе итерации команда столкнулась с рядом трудностей:

- координация работы между всеми участниками потребовала больше внимания, чем ожидалось;
- возникали сложности при связке фронтенда и бэкенда;
- настройка окружения заняла дополнительное время;
- проекту пока не хватает более формализованной внутренней документации;
- алгоритм оценки стойкости пароля требует дальнейшего улучшения;
- необходимо уделить больше внимания валидации и общей устойчивости логики;
- полезно дополнительно проработать безопасность обрабатываемых данных и UX.

При этом большая часть этих сложностей была выявлена своевременно и уже может быть использована как основа для планирования следующей итерации.

## Выводы по командной работе
Командная работа во второй итерации в целом была оценена участниками положительно. В качестве сильных сторон были отмечены:

- постоянная связь внутри команды;
- оперативное уточнение задач;

- понятное распределение ответственности.

Именно за счёт координации и последовательного контроля задач удалось довести результаты итерации до общего рабочего состояния.
