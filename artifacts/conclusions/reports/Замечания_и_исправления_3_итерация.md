# Замечания по текущему состоянию проекта и задачи на исправление в 3 итерации

## Общая информация
По итогам анализа текущего состояния проекта были выявлены несколько технических и организационных замечаний, которые важно учесть в рамках 3 итерации. На текущем этапе сервис уже содержит рабочее MVP, однако часть решений требует доработки для повышения корректности логики, устойчивости интерфейса и качества тестирования.

## Выявленные замечания

### 1. Изменение пользовательского пароля перед проверкой
В текущей реализации перед началом проверки используется удаление пробелов по краям строки (`strip`). Это приводит к изменению исходного значения пароля пользователя до анализа и до проверки по базе утечек.

Почему это важно:
- пароль должен анализироваться как точное значение, введённое пользователем;
- текущее поведение может исказить результат проверки;
- это особенно критично для проверки по базе утечек, где требуется точное совпадение хэша.

Что требуется исправить:
- убрать изменение исходного пароля перед основной проверкой;
- отдельно обработать только случай полностью пустого ввода без изменения валидных значений.

### 2. Определение факта утечки через текстовое сообщение
Во frontend-части и в тестах наличие пароля в базе утечек сейчас определяется по тексту причины, содержащему слово «утечка». Это создаёт хрупкую зависимость логики интерфейса и тестов от формулировки текстового ответа.

Почему это важно:
- при изменении текста причины отображение результата может стать некорректным;
- тесты в таком виде проверяют не контракт API, а конкретную фразу;
- логика результата должна быть устойчивой к редактуре текста.

Что требуется исправить:
- добавить в API явный флаг, например `breached: true/false`;
- перевести frontend и тесты на использование этого поля вместо анализа текста.

### 3. Автотесты зависят от вручную запущенного локального сервера
Текущие автотесты обращаются к локальному адресу `http://127.0.0.1:8000/api/check` и требуют, чтобы backend был заранее запущен вручную.

Почему это важно:
- тесты не являются полностью автоматизированными;
- такой подход затрудняет повторяемую проверку и интеграцию в CI;
- при отсутствии запущенного сервера тесты завершаются ошибкой соединения.

Что требуется исправить:
- перейти на более автономный формат тестирования;
- использовать тестовый клиент FastAPI или отдельную тестовую конфигурацию;
- разделить smoke-проверки и реальные автоматические тесты.

### 4. Логика тестов зависит от текстовых причин ответа
В текущем наборе тестов статус утечки определяется через поиск слова «утечек» в массиве причин. Это дублирует описанную выше проблему и снижает надёжность тестового покрытия.

Что требуется исправить:
- синхронизировать тесты с финальным контрактом ответа API;
- проверять структурированные поля ответа, а не текстовые формулировки.

### 5. Ограничение максимальной длины объявлено, но не используется
В конфигурации проекта уже задано ограничение максимальной длины пароля, однако в основной логике проверки оно фактически не применяется.

Почему это важно:
- между конфигурацией и поведением сервиса возникает несоответствие;
- обработка граничных случаев остаётся неполной;
- это снижает предсказуемость логики проверки.

Что требуется исправить:
- добавить фактическую проверку максимальной длины в алгоритм;
- вернуть пользователю понятное сообщение при превышении ограничения.

## Приоритеты исправлений в 3 итерации
В рамках 3 итерации целесообразно выделить следующий порядок исправлений:

1. Исправить обработку входного значения пароля без изменения исходной строки.
2. Изменить контракт API и добавить явный флаг наличия в базе утечек.
3. Перевести frontend на работу с новым полем ответа.
4. Переработать тесты так, чтобы они проверяли структуру ответа API, а не текст сообщений.
5. Добавить полноценную обработку максимальной длины и других граничных случаев.

## Заключение
На текущем этапе проект находится в рабочем состоянии и уже позволяет демонстрировать ключевой сценарий использования. Однако выявленные замечания необходимо учесть в 3 итерации, чтобы улучшить качество реализации, устранить хрупкие зависимости в логике и повысить надёжность тестирования. Исправление этих пунктов позволит перейти от базового MVP к более устойчивой и технически аккуратной версии сервиса.
